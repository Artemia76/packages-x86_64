Little lexicon of the NuTyX user's

Cards

Cards which means 'Create, Add, Remove and Download System' packages) is a set of utilities to manage your installed softwares on your NuTyX-linux.

Cards can manage dependencys for compilation as well. It can handle runtime dependencys on it's own. It's means that in 95 % of the cases, dependencies for a working app do not need to to be specified in the construction recipe.

A package

A package is an archive containing a set of files. Thoses files can be from various types. We can find libraries, command line applications, graphical users interface commands configuration information, etc

When the package is installed on your NuTyX, it will be handle by the packages manager cards

A dependency

A dependency is a package AAA needed for the runtime / for the compilation of another package BBB).

A good example of dependency is the gtk3 package which is used by many packages of the gnome collection.

A collection

A collection is collection all the packages that brings a set of fonctionnalities.

When you install the base system from the MINI ISO or from the script, you installed the base collection and partielly the base-extra collection The gui collection for example allows you to install a graphical user interface. On this graphical user interface, we will choose our favorite windows manager.

Under NuTyX, its possible to install kde5, gnome, mate, xfce4, lxde collections,

A group

Under NuTyX, a group of packages means a set of packages that the begining of the name is common: A exemple of group of packages under NuTyX:

  • gstreamer1
  • gstreamer1-plugins-bad
  • gstreamer1-plugin-good
  • gstreamer1-plugin-libav
  • gstreamer1-...

    • Normaly, a group of packages will always be in the same collection

    A category

    A category define a set of applications having similar fonctionalities: A typical example of category is "Internet" which include browsers, irc and mails clients,etc.

    The base system

    The base system of NuTyX is made of a set of a minimum number of packages that will never be removed.

    The list of packages to keep when we want to return to a base system is define in the /etc/cards.conf file. The keyword at the begining of the line is: base.

    A script

    The script is used, like in cinema, to define each action that have to be executed by the interpreter. In this case, the script will be then the set of organised actions to do for the construction of a package.

    A script can also be used for installing a NuTyX from another linux distribution or from another already installed NuTyX.

    A recipe

    A recipe define the construction of a package. Under NuTyX, the recipe of a package is located in the Pkgfile file.

    A port

    In NuTyX, a port is made of a folder including all the necessary files for the construction of a package. It will always at least contains a file named Pkgfile.

    The port can also, in somes cases, contains a pre-install or a post-install script. It can also contains a README file describing various options, actions, configurations of the port.

    The term port make reference to a remote depot containing ports and use a client programm able to downloads thoses ports from this remote depot.

    NuTyX use the ports tool for downloading all the ports from this remote depot and locate them in the /usr/ports/<name_of_collection> folder

    The 'Pkgfile' file

    Pkgfile is a text file. It contains the recipe of construction of an application. The result of the construction will be a package which can be installed with the packages manager cards

    The contents of this file follow a standard schema and is compose of three parts:

    1. General informations (optionnal but highly recommended) about the package: Description, URL, Developer(s), Packager(s), dependencys
    2. Mandatory information for the construction of the package: Name, version, release
    3. The "build" function define the construction itself of the package

    The Pkgfile file must be located in a folder with the same name as the name variable define in this Pkgfile file.

    Example of a such Pkgfile located in the mypackage folder:

    # Depends on: glibc
description="My first package"
url="http://downloads.nutyx.org"
packager="tnut at nutyx dot org"
name=mypackage
version=1.0
release=1
source=(http://downloads.nutyx.org/files/$name-$version.tar.xz)
build() {
cd ${name}-$version
./configure --prefix=/usr \
--disable-static
make
make DESTDIR=$PKG install
}

    A scenario

    What does the term 'Scenario' means in a NuTyX installation context ?

    Don't search Google for "scenario" as Linux terminology (... not just yet anyway?), I suspect my formula doesn't exist yet on other Linux distributions .

    This is an idea I formulated while I was creating the cards packages manager.

    Over the years, I've experienced an extensive and growing target audience desiring to use a distribution similar to what NuTyX offers at present. The same audience indicated to me that they wanted to build their own kernel using their own preferential style and package mix.

    Most new-comers pursuing the “building infrastructure scenario of NuTyX” are desiring, just like the preceding audience, to compile their own downloaded packages of choice at their leisure. They want flexibility and options in doing their compilation. So this is where the concept of "scenario" comes to the rescue. Some will find the default packaged collection not suited to their preference and wish to modify it. Some users will want to mix their standard available binary packages with their own ported packages. Others wanting to compile packages in an isolated environment, whereby keeping their original system intact and isolated. These are just some of the different requests for building their system.

    I describe these different options of installation as “scenario's” that is accessible within the framework of NuTyX.

    A chroot

    A chroot able to isolate our working environment where we will be able to build all the recipes coming from various ports.

    A chroot is needed specialy to assure that none dependencys will be installed from the host operating system.

    Levels

    When cards compile a package, it needs to know the level of compilation of this package.

    Each package may or may not contains dependencys.

    If a package named AAAA do contains any dependencys, it will be at the first level means level 0 (zero).

    If a package named BBBB contains a dependency named AAAA, this package will be at second level, means level 1.

    The list of dependencys for the compilation of a package is define at the line:

    # Depends on: glib atk pango gtk xorg-libx11

    of a Pkgfile.
