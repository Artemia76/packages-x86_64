Little lexicon of the NuTyX user's

Cards

Cards which means 'Create, Add, Remove and Download System' packages) is a set of utilities to manage your installed softwares on your NuTyX-Linux.

Cards can manage dependencies for compilation as well. It can handle runtime dependencies on it's own. This means that in 95 % of all cases, dependencies for a working app do not need to to be specified in the construction recipe.

A package

A package is an archive containing a set of files. Thoses files can be from various types. We can find libraries, command line applications, graphical user interfaces, commands, configuration information, etc

When the package is installed on your NuTyX, it will be handled by the package manager cards

A dependency

A dependency is a package AAA needed for the runtime / for the compilation of another package BBB).

A good example of a dependency is the gtk3 package which is used by many packages of the gnome collection.

A collection

A collection is a collection of all the packages that bring a set of functionalities.

When you install the base system from the MINI ISO or from the script, you installed the base collection and partielly the base-extra collection The gui collection for example allows you to install a graphical user interface. For such a graphical user interface, we will choose our favorite windows manager.

Under NuTyX, it's possible to install kde5, gnome, mate, xfce4, lxde desktop collections,

A group

Under NuTyX, a group defines a set of packages which begin with a common name. Additional parts are separated by a 'hyphen-minus' (-).

An example of a NuTyX package group:

  • gstreamer1
  • gstreamer1-plugins-bad
  • gstreamer1-plugin-good
  • gstreamer1-plugin-libav
  • gstreamer1-...

    • Normaly, a group of packages will always be in the same collection

    A category

    A category defines a set of applications which have similar functionalities: A typical example of category is "Internet" which includes browsers, irc and mail clients, etc.

    The base system

    The NuTyX 'base system' defines the minimum set of packages that will never be removed.

    The list of packages to keep when we want to return to a base system is defined in the /etc/cards.conf file. The keyword at the begining of the line is: base.

    A script

    The script is used, like in cinema, to define each action that has to be executed by the interpreter. In this case, the script will be a set of organised actions to be executed for the construction of a package. See recipe

    A script can also be used for installing a NuTyX-Linux from another linux distribution or from another already installed NuTyX-Linux.

    A recipe

    A recipe is a script which defines the construction of a package. Under NuTyX, the recipe of a package is located in the Pkgfile file.

    A port

    In NuTyX, a port is a folder which contains all necessary files for the construction of a package. It will at least contain a file named Pkgfile.

    The port can also, in somes cases, contain a 'pre-install' or a 'post-install' script. It can also contain additionally a README file describing various options, actions, configurations of the port or other required files e.g. patch files..

    The term port makes reference to a remote depot containing ports which can be downloaded with a suitable client programm.

    NuTyX use the ports tool for downloading all the ports from a remote depot and store them by default in the /usr/ports/<name_of_collection> folder.

    The 'Pkgfile' file

    A Pkgfile is a script file. It contains the recipe for the construction a package which can be installed with the package manager cards

    The content of this file follows a common schema and is compose of three parts:

    1. `Pkgfile-Header`: (optional but highly recommended) general information about the package: Description, URL, Developer(s), Packager(s), dependencies
    2. `Mandatory information`: for the construction of the package: Name, version, release
    3. `Required build function`: The "build" function defines the construction itself of the package

    NOTE: the above outlines just the most general situation.

    The Pkgfile file must be located in a folder with the same name as the name variable defined in the Pkgfile.

    Example of a Pkgfile located in the mypackage folder:

    # Depends on: glibc
description="My first package"
url="http://downloads.nutyx.org"
packager="tnut at nutyx dot org"
name=mypackage
version=1.0
release=1
source=(http://downloads.nutyx.org/files/$name-$version.tar.xz)
build() {
cd ${name}-$version
./configure --prefix=/usr \
--disable-static
make
make DESTDIR=$PKG install
}

    A scenario

    What does the term 'Scenario' mean in a NuTyX installation context ?

    Don't search Google for "scenario" as Linux terminology (... not just yet anyway?), I suspect my formula doesn't exist yet on other Linux distributions .

    This is an idea I formulated while I was creating the cards packages manager.

    Over the years, I've experienced an extensive and growing target audience desiring to use a distribution similar to what NuTyX offers at present. The same audience indicated to me that they wanted to build their own kernel using their own preferential style and package mix.

    Most new-comers pursuing the “building infrastructure scenario of NuTyX” are desiring, just like the preceding audience, to compile their own downloaded packages of choice at their leisure. They want flexibility and options in doing their compilation. So this is where the concept of "scenario" comes to the rescue. Some will find the default packaged collection not suited to their preference and wish to modify it. Some users will want to mix their standard available binary packages with their own ported packages. Others wanting to compile packages in an isolated environment, whereby keeping their original system intact and isolated. These are just some of the different requests for building their system.

    I describe these different options of installation as “scenario's” that is accessible within the framework of NuTyX.

    A chroot

    A chroot is able to isolate our working environment so that we are able to build all the recipes coming from various ports.

    It is also needed to assure that no dependency will be installed from the host operating system.

    Levels

    When cards compiles a package, it needs to know the level of compilation of this package.

    Each package may or may not contains dependencies.

    If a package named AAAA contains some dependencies, it will be at the first level: level 0 (zero).

    If a package named BBBB contains a dependency named AAAA, this package will be at second level: level 1 (one).

    The list of dependencies for the compilation of a package is defined by the line:

    # Depends on: glib atk pango gtk xorg-libx11

    in a Pkgfile.
